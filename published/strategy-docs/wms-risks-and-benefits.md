<div class="banner banner-blue"><strong>Reference implementation note:</strong> Where specific tools are named (Claude Desktop, Cursor, Claude Code), these refer to the reference implementation only. The WMS architecture is tool-agnostic — the three roles can be filled by your organisation's existing AI stack. The <strong>Tool Mapping Guide</strong> shows equivalent configurations for Microsoft, Google, GitHub-centric, and internal LLM deployments. Risks and benefits relating to specific tools apply to those tools only, not to the WMS methodology itself.</div>

<tr data-level="high"><td>🔴</td><td><span class="cat-tag">Adoption</span></td><td><strong>Partial adoption is invisible and produces no value.</strong> A practitioner who skips the Compound Step twice, runs two sessions without a cleanup, and edits <code>dashboard.json</code> directly once has broken the system in ways that are not immediately visible — but the SKILL library is thin, the audit chain has gaps, and the state machine is unreliable.</td><td>Establish the full ritual in the first four sessions and do not shorten it. <tr data-level="high"><td>🔴</td><td><span class="cat-tag">Adoption</span></td><td><strong>The Compound Step is the first thing that gets dropped under time pressure.</strong> It's the last step of the session, it's optional-feeling, and its payoff is diffuse and delayed. Every session that ends without it is a session where institutional memory was not captured.</td><td>Build the Compound Step into the session close as a non-negotiable 10-minute block, not an optional add-on. Frame it explicitly: "the value of next month's sessions depends on whether this session's knowledge is captured today."</td></tr>
<tr data-level="high"><td>🔴</td><td><span class="cat-tag">Technical</span></td><td><strong>Claude Desktop can write to governed files if the project instructions are imprecise or overridden.</strong> The "never writes governed files" constraint is behavioural, not architectural. A single direct write to <code>dashboard.json</code> that bypasses the briefing mechanism creates an untracked change with no audit trail.</td><td>State the constraint explicitly in project instructions with the consequence: "If you write directly to a governed file, revert the change immediately and produce a briefing instead." The pre-commit hook catches this at commit time, not at write time.</td></tr>
<tr data-level="high"><td>🔴</td><td><span class="cat-tag">Organisational</span></td><td><strong>Enterprise tool approval timelines can delay deployment by months.</strong> Claude Desktop, Cursor, and Claude Code may require IT security review, procurement approval, and data handling assessment before they can be used on company devices.</td><td>Run Spec A only (no Spec B) during the approval period. Spec A requires only a text editor and one AI chat tool — almost certainly already approved. Begin building the SKILL library and task discipline immediately; the infrastructure can catch up.</td></tr>
<tr data-level="high"><td>🔴</td><td><span class="cat-tag">Adoption</span></td><td><strong>The system degrades gracefully in ways that look like it's still working.</strong> Sessions continue, tasks appear to progress, the dashboard renders — but without the SKILL library growing, without clean audit trails, and without verifiable <code>done_when</code> criteria being enforced, the WMS is producing the appearance of governance without the substance.</td><td>The Day 5 cold-run test (second person runs a session from the files alone) is the primary diagnostic. Run it at week one, at month one, and at every phase transition. If the second person needs to ask questions, the system has degraded.</td></tr>
<tr data-level="high"><td>🔴</td><td><span class="cat-tag">Strategic</span></td><td><strong>The specification's transparency creates IP exposure.</strong> Spec A and Spec B are readable documents. A client who has received them — or a competitor who encounters them — can replicate the infrastructure without AIValueWorx.</td><td>The infrastructure is not the IP. The SKILL library, the pre-populated task library with validated <code>done_when</code> criteria, and the pattern recognition that produces good briefings — these are not in the spec and cannot be replicated without doing the engagements. The spec is the scaffolding; the knowledge is the value. Position this explicitly.</td></tr>
<tr data-level="high"><td>🔴</td><td><span class="cat-tag">Commercial</span></td><td><strong>Client self-sufficiency reduces ongoing engagement revenue.</strong> A successful engagement produces a client who can operate independently. That is the stated goal — and it means the client has less reason to maintain an active commercial relationship.</td><td>Design the ongoing relationship as advisory augmentation: new task library phases as the methodology evolves, annual SKILL library reviews, phase transition support, second-workstream pilots. The client who is genuinely capable values this more than the client who is dependent.</td></tr>
<tr data-level="high"><td>🔴</td><td><span class="cat-tag">Methodology</span></td><td><strong>Phase sequencing violations are the most common enterprise failure mode.</strong> Pressure from leadership to "just start doing AI things" before governance exists leads to Phase 3 pilots running without Phase 2 foundations — then Phase 4 produces results that cannot be acted on safely.</td><td>The task dependency chain in the pre-populated library is the primary defence. Make the dependency logic explicit to the sponsor at engagement start: "we cannot run the pilot in a way that scales until these governance tasks are complete. Here is exactly what those tasks require."</td></tr>
<tr data-level="high"><td>🔴</td><td><span class="cat-tag">Organisational</span></td><td><strong>Phase 4 stop decisions are politically difficult to execute.</strong> A pilot that should be stopped will often be continued because stopping requires the sponsor to acknowledge that the investment did not produce the expected result.</td><td>Frame the stop decision criteria at Phase 3 start, not at Phase 4 close. When the metrics and stop thresholds are agreed before the pilot runs, the Phase 4 decision becomes a measurement outcome rather than a judgment call. Document this framing in the Phase 2 governance tasks.</td></tr>
<tr data-level="medium"><td>🟡</td><td><span class="cat-tag">Technical</span></td><td><strong>SKILL library discovery degrades at scale.</strong> At 10 SKILL files, the design tool loads them all and operates correctly. At 50+, the context window fills and the selection of which files to load becomes implicit and unreliable. Relevant SKILL files stop being loaded; governance inputs become inconsistent.</td><td>Domain tags in SKILL file YAML front matter are a partial solution. A more robust answer — a tagging taxonomy and a session-start prompt that filters by domain — needs to be built before the library exceeds 30 files. This is a known gap in the current specification.</td></tr>
<tr data-level="medium"><td>🟡</td><td><span class="cat-tag">Technical</span></td><td><strong><code>dashboard.json</code> merge conflicts in team use are recoverable but disruptive.</strong> Two collaborators modifying the same task state in the same session will produce a JSON merge conflict that requires manual resolution.</td><td>Pull before every session, push at every session close, and coordinate session windows verbally in small teams. For larger teams, a lightweight session booking convention (who is working on which workstream today) eliminates most conflicts before they occur.</td></tr>
<tr data-level="medium"><td>🟡</td><td><span class="cat-tag">Technical</span></td><td><strong>Pre-commit hook silently fails on some configurations.</strong> If <code>validate_dashboard.py</code> is not executable, if Python 3 is not on the PATH at commit time, or if the hook was installed incorrectly, commits will proceed without validation. The failure is silent.</td><td>Verify the hook explicitly after installation: stage a deliberately invalid <code>dashboard.json</code> and confirm the commit is blocked. Run this verification again after any machine rebuild or Python update.</td></tr>
<tr data-level="medium"><td>🟡</td><td><span class="cat-tag">Adoption</span></td><td><strong>The validator gets bypassed when it blocks an urgently needed commit.</strong> <code>--no-verify</code> exists as a Git flag and developers know how to use it. A single bypassed validation is a single untracked inconsistency; a habit of bypassing is the end of governance-as-code.</td><td>Log every <code>--no-verify</code> commit in the COMPOUND.md decisions register with the reason. This doesn't prevent bypasses but makes them visible and creates a record that can be reviewed. If bypasses are frequent, the validator rules need loosening — not disabling.</td></tr>
<tr data-level="medium"><td>🟡</td><td><span class="cat-tag">Commercial</span></td><td><strong>Tool vendor dependency concentrates risk.</strong> The reference implementation depends on Anthropic (Claude Desktop, Claude Code), Cursor, and GitHub. The SKILL library and task library are vendor-independent.</td></tr>

<tr data-level="medium"><td>🟡</td><td><span class="cat-tag">Organisational</span></td><td><strong>Staff turnover degrades the system faster than a solo practitioner's departure.</strong> When the person who understands the session ritual leaves, their replacement may not know the system well enough to maintain it, and the knowledge layer stops growing.</td><td>The cold-run test is the leading indicator: if a new team member cannot run a session from the files alone within their first week, the documentation layer is insufficient. <tr data-level="medium"><td>🟡</td><td><span class="cat-tag">Adoption</span></td><td><strong>Session ritual shortcuts accumulate into structural problems.</strong> Skipping the start ritual once is harmless.
